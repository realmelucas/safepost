# 守岗 SafePost 项目立项书

## 1. 项目名称

守岗 SafePost：车厢盲区安全管理系统 / 物流孤岗作业安全系统。

## 2. 立项背景

物流转运中心、仓储月台和车厢装卸作业长期存在监控盲区、单人作业、高温作业、外包人员管理不足等问题。真实事故已经证明，当作业人员在车厢内失去行动能力时，如果现场无人确认状态，事故可能在 20 分钟甚至更长时间内无人发现。

守岗 SafePost 立项的直接目标，不是做一款智能硬件，而是建立一套低成本、可部署、可追溯的失联发现机制，避免“人已经异常，但系统无人知道”的情况再次发生。

## 3. 项目愿景

让每一位进入车厢作业的人，都能被看见、被确认、被守护。

## 4. 项目目标

- 上岗前可筛查：通过身份、血压、酒精检测降低准入风险。
- 作业中可监护：持续确认人员在线、运动状态和响应状态。
- 异常时可发现：SOS、静止、失联、无响应自动报警。
- 事故后可追溯：形成检测、作业、报警和处置完整记录。

## 5. MVP 目标

第一阶段不做 AI，不做 GPS，不做摄像头，不做心率和血氧监测。

MVP 验证链路：

进入车厢 → 开始作业 → 模拟失去行动能力 → 无响应 → 系统自动发现 → 企业微信报警 → 主管 2 分钟内确认异常。

## 6. 项目范围

### 6.1 第一版必须完成

- 工牌唯一 ID。
- SOS 报警。
- 静止检测。
- 无响应检测。
- 作业计时。
- 喝水提醒。
- 基站接收。
- 后台显示。
- 企业微信报警。

### 6.2 第一版暂不完成

- AI 分析。
- GPS。
- 摄像头。
- 心率监测。
- 血氧监测。
- 皮带机自动联动。

## 7. 技术方案

| 层级 | 技术 |
| --- | --- |
| 工牌 | ESP32-C3、安全按键、三色状态灯、蜂鸣、震动、加速度计 |
| 基站 | ESP32-S3、Wi-Fi/以太网、BLE/ESP-NOW/MQTT |
| MQTT | EMQX |
| 后端 | Node.js + NestJS |
| 数据库 | PostgreSQL |
| 后台 | React + Vite + Ant Design |
| 通知 | 企业微信机器人或应用消息 |

## 8. 项目组织

| 角色 | 职责 |
| --- | --- |
| 产品负责人 | 需求定义、试点流程、客户反馈 |
| 硬件工程师 | 工牌和基站原型、外壳、续航、可靠性 |
| 嵌入式工程师 | 工牌/基站固件、通信、低功耗 |
| 后端工程师 | API、MQTT 接入、报警规则、日志 |
| 前端工程师 | 管理后台、看板、报警处置 |
| 现场实施 | 试点部署、培训、设备维护 |

## 9. 成功标准

- SOS 到企业微信通知小于 10 秒。
- 无响应异常 2 分钟内被主管确认。
- 工牌在线率大于 98%。
- 关键报警日志完整率大于 99%。
- 一线工人可以在 5 分钟内理解红黄绿灯和 SOS 操作。
- 试点客户愿意继续扩大部署或进入付费验证。

## 10. 主要风险

| 风险 | 应对 |
| --- | --- |
| 工牌忘戴或代戴 | 上岗绑定、现场抽检、后续接入门禁 |
| 无线信号遮挡 | 多基站部署、RSSI 监测、离线缓存 |
| 误报过多 | 分级报警、规则调参、冷却时间 |
| 工牌续航不足 | 低功耗策略、充电柜、低电量提醒 |
| 现场抵触 | 工牌轻量化、培训简化、强调救命价值 |

## 11. 立项结论

守岗 SafePost 聚焦明确的高风险场景，MVP 验证链路清晰，技术路径可实现，内部管理价值明确。建议先作为内部安全项目进入原型开发和现场试点准备阶段，待失联发现机制被验证后，再评估单独立项和商业化可能性。
